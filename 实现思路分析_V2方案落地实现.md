# V2 方案落地实现思路分析

> 本文针对 `思路V2_引入PyOD的质检策略Agent实施方案.md` 中规划的"规则引擎 + LLM 单 Agent + PyOD 异常检测（双通道）"架构，分析**如何实现**：分层设计、关键数据结构、核心算法、代码骨架与实施顺序。目标是让开发可以按本文直接动手编码。

---

## 一、总体实现策略

### 1.1 分层与依赖方向

```
api 层（FastAPI 路由）
   ↓ 只依赖
pipeline 层（audit_pipeline：编排 + 融合后处理）
   ↓ 只依赖
service 层（preprocess / rule_engine / evidence / feature / anomaly / llm_agent，互相独立）
   ↓ 只依赖
schema 层（Pydantic 模型）+ knowledge 层（YAML 配置，启动时加载为不可变对象）
```

实现纪律：

- **service 层各模块之间零依赖**，只通过 pipeline 传递 schema 对象通信。这保证规则引擎、异常检测、LLM 三路信号可独立开发、独立测试、独立降级。
- **所有业务知识进 YAML，不进 Python**。Python 只实现"匹配机制"，规则内容（关键词、分支条件、风险等级）全部配置化，规则迭代不发版代码。
- **纯函数优先**：preprocess、rule_engine、evidence、融合后处理全部实现为无状态纯函数（输入 → 输出），只有 llm_agent（外部 API）和 anomaly_service（模型工件）持有资源句柄。

### 1.2 技术选型落地

| 组件 | 选型 | 理由 |
|---|---|---|
| Web 框架 | FastAPI + Uvicorn | 原生 Pydantic 集成、async 支持（LLM 调用 IO 密集） |
| 规则匹配 | 标准库 `re` + 自实现条件组合器 | 规则量级（首期 ~50 条）无需引入 Drools 类引擎；正则 + AND/OR/NOT 组合足够 |
| LLM 调用 | OpenAI 兼容 SDK（`openai` 包，base_url 可配） | 国内主流模型（通义/DeepSeek/Kimi 等）均兼容该协议，换模型只改配置 |
| 结构化输出 | JSON mode + Pydantic 二次校验 + 失败重试 | 双保险，不信任模型自觉 |
| 异常检测 | pyod（IForest/ECOD/KNN）+ ADEngine（离线） | 见 V2 方案第二、六节 |
| Embedding | 可配置：本地 sentence-transformers 或 HTTP API | `.env` 切换，接口抽象为 `embed(texts) -> np.ndarray` |
| 工件序列化 | joblib + manifest.json | 检测器、缩放器、降维器、阈值打包成带版本目录 |

## 二、关键数据结构（schema 层先行）

整个系统围绕三个核心 Pydantic 模型流转，先于一切编码固化：

```python
# app/schemas/audit.py（核心摘录）

class Turn(BaseModel):
    speaker: Literal["agent", "customer"]   # 预处理统一 left→agent, right→customer
    index: int                              # 轮次序号，证据回溯用
    text: str

class RuleHit(BaseModel):
    rule_id: str
    category: str                           # 一级场景：引流第三方平台/贷款相关/...
    subtype: str                            # 二级子类：引导用户主动添加微信/...
    risk_level: Literal["合规", "低风险", "中风险", "高风险"]
    fraud_flag: bool                        # 是否涉诈类规则
    priority: int                           # 就高不就低排序用
    matched_turns: list[int]                # 命中的轮次 index
    matched_text: list[str]                 # 触发匹配的原文片段

class AnomalyResult(BaseModel):
    enabled: bool
    score: float | None = None              # rank-normalized 共识分 [0,1]
    is_outlier: bool | None = None
    detectors: list[DetectorScore] = []
    top_deviating_features: list[str] = []
    model_version: str | None = None

class AuditResponse(BaseModel):
    is_violation: bool
    is_fraud: bool
    violation_type: str | None
    risk_level: Literal["正常", "低风险", "中风险", "高风险"]
    summary: str                            # "类型：简要分析" 格式
    explanation: str
    evidence: list[str]
    rule_hits: list[RuleHit]
    needs_human_review: bool
    review_reason: Literal["none", "rule_hit_fraud", "rule_llm_conflict",
                           "low_confidence", "anomaly_only"]
    confidence: float
    anomaly: AnomalyResult
```

中间传递对象 `PipelineContext`（dataclass）贯穿管线，每个 service 只读上游字段、写自己的字段，便于调试时 dump 全量中间态。

## 三、预处理实现（preprocess.py）

输入是规范中那种 `left:...right:...` 粘连的 ASR 文本，实现要点：

1. **说话人切分**：正则 `re.split(r'(left:|right:)', raw)` 切流，处理规范案例中常见的"一行内多次切换说话人"（如 `right:冬天left:今年是没有`）；left→agent（外呼侧）、right→customer。
2. **噪声清理**：去除 `\*` 脱敏符周边空白、全角/半角统一、连续重复语气词压缩（"嗯嗯嗯"→"嗯"，保留一份证据原文映射避免证据引用失真——清理只用于匹配，证据展示用原文切片）。
3. **轮次构建**：连续同 speaker 文本合并为一个 `Turn`，编号 index。
4. **双文本视图**：产出 `normalized_turns`（匹配用，经同义词归一）与 `raw_turns`(证据用，保留原文)，两者 index 对齐。

同义词归一实现：启动时把 `normalization.yaml` 编译成一个 `dict[str, str]`（变体→标准词）+ 单次构建的 `re.Pattern`（`|`.join 全部变体，按长度降序避免短词抢占），`normalize(text)` 一次 `re.sub` 完成。这一步让下游规则和特征统计只需面对标准词（如"企微/工作微信/企业微信"统一为"企业微信"）。

## 四、规则引擎实现（rule_engine.py）——本系统最核心的工程难点

### 4.1 规则 YAML 的表达能力设计

规范的难点是"条件组合 + 角色方向 + 上下文例外"（如加微信三分支）。YAML schema 设计为**模式组 + 布尔组合 + 说话人约束**：

```yaml
# app/knowledge/chongqing_rules.yaml（示例条目）
- rule_id: CQ-DIVERT-WECHAT-USER-INITIATED
  category: 引流第三方平台
  subtype: 引导用户主动添加微信
  risk_level: 高风险
  fraud_flag: false
  priority: 80
  match:
    all:                                  # AND
      - speaker: agent                    # 只匹配外呼侧话术
        any_pattern:                      # OR，正则列表
          - "你(打开|点开).{0,4}微信"
          - "(添加|搜索|加一?下).{0,8}(微信|好友|通讯录)"
          - "点击.{0,4}(加号|添加朋友)"
      - speaker: agent
        any_pattern:
          - "(报|输入|搜索).{0,6}(号码|手机号)"
    none:                                 # NOT（例外条件）
      - speaker: agent
        any_pattern:
          - "(我|这边|我们)(主动|待会儿?|等下)?加(您|你)"   # 外呼方主动加→不命中本规则
  window: 6        # all 条件需在 6 个轮次窗口内共同满足
  decision_notes: "引导用户主动操作添加微信，不区分场景均为高风险"

- rule_id: CQ-DIVERT-WECHAT-AGENT-INITIATED
  category: 引流第三方平台
  subtype: 外呼人员主动添加用户微信
  risk_level: 合规
  priority: 10
  match:
    all:
      - speaker: agent
        any_pattern: ["(我|这边|经理|专员).{0,6}加(您|你).{0,4}微信"]
    none:
      - speaker: agent
        any_pattern: ["你(打开|点开).{0,4}微信", "点击.{0,4}加号"]
```

要点：

- **`all` / `any` / `none` 三键布尔组合**覆盖规范全部条件形态；嵌套不超过两层，避免 DSL 复杂化。
- **`speaker` 约束**直接解决"主动方是谁"问题——规范中合规/违规的分水岭大多在于动作出自外呼侧还是用户侧、指令的施动方向。
- **`window` 滑动窗口**解决"多条件需共现"问题（如"提到微信"+"报号码引导搜索"要在邻近轮次同时出现才算引导操作），防止跨整通通话的远距离误组合。
- **合规规则也显式建模**（priority 低），用于在 LLM 上下文中明确告知"该行为已被判定合规"，抑制模型把合规加微信误判违规——这是对规范例外逻辑最直接的工程化。

### 4.2 匹配算法

```python
def evaluate(turns: list[Turn], rules: list[CompiledRule]) -> list[RuleHit]:
    hits = []
    for rule in rules:                          # 规则量 ~50，逐条全扫即可，无需索引
        for start in range(len(turns)):
            window = turns[start : start + rule.window]
            if _match_group(window, rule.all_, mode="all") and \
               not _match_group(window, rule.none_, mode="any"):
                hits.append(_build_hit(rule, window))
                break                           # 每规则记首个命中窗口即可
    return hits
```

- 启动时把 YAML 编译为 `CompiledRule`（正则 `re.compile` 预编译），请求期零编译开销；
- 单通话 50 条规则 × 滑窗匹配，纯正则在毫秒级，性能不构成问题；
- `_build_hit` 记录命中的轮次 index 与触发片段（从 raw_turns 取原文），直接喂证据提取。

### 4.3 就高不就低合并（merge.py 内联在 rule_engine）

```python
RISK_ORDER = {"合规": 0, "低风险": 1, "中风险": 2, "高风险": 3}

def merge_hits(hits: list[RuleHit]) -> RuleVerdict:
    violations = [h for h in hits if h.risk_level != "合规"]
    if not violations:
        return RuleVerdict(candidate="正常", hits=hits)
    top = max(violations, key=lambda h: (RISK_ORDER[h.risk_level], h.priority))
    return RuleVerdict(
        candidate=top.category,
        candidate_risk=top.risk_level,
        fraud_candidate=any(h.fraud_flag for h in violations),
        hits=hits,                               # 全量命中保留给 LLM
    )
```

注意：合并结果只是**候选结论**，最终裁决在 LLM；但涉诈规则命中时 `fraud_candidate=True` 会在融合后处理中强制 `needs_human_review=True`（代码兜底，不依赖 LLM）。

## 五、证据提取实现（evidence_extractor.py）

输入 RuleHit 的 `matched_turns`，输出展示级证据：

1. 以命中轮次为中心取 `±1` 轮上下文窗口；
2. 从 **raw_turns**（未归一的原文）切片，保证证据与录音转写逐字一致（规范的人工复核要求）；
3. 片段超长（规范案例单轮可达数千字）时，以命中片段为锚点截取前后各 ~80 字，首尾加省略号；
4. 去重：多规则命中同一轮次只输出一份片段，但在 RuleHit 中各自保留 index 引用。

## 六、特征工程实现（feature_extractor.py）

### 6.1 接口契约（在线/离线共用，杜绝训练-推理偏斜的关键）

```python
class FeatureExtractor:
    def __init__(self, spec: FeatureSpec, embedder: Embedder): ...
    def extract(self, ctx: PipelineContext) -> np.ndarray:     # 单条，在线用
    def extract_batch(self, ctxs: list[PipelineContext]) -> np.ndarray:  # 离线用
    @property
    def feature_names(self) -> list[str]:        # 与向量列严格对齐，解释偏离特征用
```

- 行为特征的词表与口径在 `feature_spec.yaml`，结构与规则 YAML 同风格（pattern 列表 + speaker 约束），实现复用规则引擎的归一与正则编译器；
- **特征顺序由 spec 文件顺序唯一确定**，spec 内容哈希写入模型 manifest，在线加载时校验哈希不一致即拒绝启动——这是防训练-推理偏斜的硬手段。

### 6.2 两路特征拼接

```python
def extract(self, ctx):
    behav = self._behavioral(ctx)        # ~30 维：词频/密度/结构比例（见 V2 方案 4.2 表）
    emb   = self.embedder.embed(ctx.agent_text)   # 外呼侧全文 embedding，384/768 维
    emb_r = self.pca.transform(emb)      # 离线拟合的 PCA，降到 32 维
    return np.concatenate([self.scaler.transform(behav), emb_r])
```

- 行为特征示例实现：`引导操作密度 = count(操作指令pattern命中) / agent轮次数`；`资金动作信号 = 资金词命中数 + 金额数字出现次数`；均为 O(文本长度) 的正则统计；
- `scaler`（RobustScaler，对长尾稳健）与 `pca` 都在离线训练时 fit，随工件序列化，在线只 transform。

## 七、PyOD 异常检测实现（anomaly_service.py）

### 7.1 离线训练（scripts/train_anomaly.py）

```python
from pyod.utils.ad_engine import ADEngine

def train(corpus_dir: Path, out_dir: Path):
    ctxs = [preprocess(load(f)) for f in corpus_dir.glob("*.json")]
    fe = FeatureExtractor.fit(spec, ctxs)          # fit scaler + pca
    X = fe.extract_batch(ctxs)

    engine = ADEngine()
    state = engine.investigate(X)                  # 全流程：画像→陷阱检查→plan→run→analyze
    report = engine.report(state, format="json")

    quality = report["quality"]                    # separation / agreement / stability
    assert quality["verdict"] == "acceptable", f"质量不达标: {quality}"

    detectors = state.fitted_detectors             # plan 选出并已 fit 的 Top-N（预期 IForest/ECOD/KNN）
    threshold = np.quantile(state.consensus_scores, 1 - TOP_K_RATIO)  # 按复核容量倒推

    save_artifacts(out_dir, detectors=detectors, extractor=fe,
                   threshold=threshold, report=report,
                   manifest={"version": ..., "spec_hash": fe.spec_hash,
                             "train_size": len(X), "trained_at": ...})
```

实现注意：

- ADEngine 的 Escalation Trigger 在金融场景会要求**至少双检测器共识**——工件中固化 `plan` 选出的全部检测器，在线打分对各检测器分数做 rank-normalize 后取均值复现共识逻辑；
- 若 ADEngine 实际 API 与上述伪码有出入（v3 仍在演进），以 `engine.plan/run/analyze` 分步形态为准做薄适配层 `ADEngineAdapter`，把对 pyod 的依赖收敛在 anomaly_service 一个文件内，保证升级 pyod 不波及管线。

### 7.2 在线打分（同步链路）

```python
class AnomalyService:
    def __init__(self, model_dir: Path | None):
        self._artifacts = load_artifacts(model_dir) if model_dir else None  # 启动加载一次

    def score(self, ctx: PipelineContext) -> AnomalyResult:
        if not self._artifacts:
            return AnomalyResult(enabled=False)
        try:
            x = self._artifacts.extractor.extract(ctx).reshape(1, -1)
            per_det = {name: float(d.decision_function(x))           # 不 fit，只打分
                       for name, d in self._artifacts.detectors.items()}
            score = rank_normalize_consensus(per_det, self._artifacts.calibration)
            return AnomalyResult(
                enabled=True, score=score,
                is_outlier=score >= self._artifacts.threshold,
                detectors=[DetectorScore(name=k, score=v) for k, v in per_det.items()],
                top_deviating_features=self._explain(x),             # 行为特征 z-score Top3
                model_version=self._artifacts.manifest["version"])
        except Exception:
            logger.exception("anomaly scoring degraded")
            return AnomalyResult(enabled=False)                      # 降级，绝不抛出
```

- `rank_normalize`：离线训练时保存训练集各检测器分数的分位数表（calibration），在线用 `searchsorted` 把原始分映射到 [0,1] 分位——单条样本无法现场 rank，必须靠训练期分布定标；
- `_explain`：只对行为特征（可解释维度）算相对训练分布的 z-score，取绝对值 Top3 的 `feature_names`，embedding 维度不参与解释；
- 整个 `score` 包 try/except，**任何失败返回 `enabled=False`**，管线行为退化为 V1。

## 八、LLM Agent 实现（llm_agent.py）

### 8.1 上下文组装（不喂原始全文之外的冗余）

Prompt 分四段拼装：

1. **系统提示词**（`chongqing_strategy_agent.md` 全文）：判定纪律 + 风险等级口径 + 异常信号使用纪律；
2. **规则知识摘要**：只注入**本次命中规则**及其同 category 的分支说明（含合规分支），而非全量规则库——控制 token，并让模型看到"例外"；
3. **Few-shot**：从 `few_shots.json` 按命中 category 选 2-3 条最相关样例（首期简单按 category 字典匹配，不引入向量检索）；
4. **本次案件**：标准化对话（超长通话截断策略：保留命中轮次 ±3 上下文 + 首尾各 5 轮）+ RuleHit 列表 + 证据片段 + 异常信号摘要（一行："统计异常分 0.87，偏离特征：补录费词频、操作指令密度。注意：异常≠违规"）。

### 8.2 结构化输出三重保障

```python
async def adjudicate(self, payload: LLMInput) -> LLMVerdict:
    for attempt in range(3):
        raw = await self.client.chat.completions.create(
            model=self.model, response_format={"type": "json_object"},
            temperature=0.1, messages=self._build_messages(payload))
        try:
            verdict = LLMVerdict.model_validate_json(raw.choices[0].message.content)
            self._sanity_check(verdict, payload)     # 见下
            return verdict
        except (ValidationError, SanityError) as e:
            payload = payload.with_repair_hint(str(e))   # 失败原因回注重试
    return LLMVerdict.fallback()                          # 三次失败→无法判定+强制人工复核
```

`_sanity_check` 代码级校验（不信任模型）：

- `evidence` 中每个片段必须是输入对话的子串（防编造证据）；
- `is_fraud=True` 时 `risk_level` 必须为高风险（规范硬规则）；
- `summary` 必须形如 `"<类型>：<分析>"`（正则校验规范要求的说明格式）；
- `violation_type` 必须属于规则库 category 枚举或 `"其他"`。

### 8.3 模型职责边界（在提示词中固化）

模型只做四件事：①在规则候选基础上细化最终类型；②判断是否升级涉诈；③生成"类型+分析"说明；④给复核建议与置信度。**不允许**：推翻"涉诈规则命中→高风险"的硬规则（即便模型想推翻，融合后处理也会纠正）。

## 九、管线编排与融合后处理（audit_pipeline.py）

```python
async def run(self, request: AuditRequest) -> AuditResponse:
    ctx = preprocess(request)                                   # 1 标准化
    rule_task = asyncio.to_thread(self.rules.evaluate, ctx)     # 2 规则（CPU）
    anomaly_task = asyncio.to_thread(self.anomaly.score, ctx)   # 4 异常打分
    hits, anomaly = await asyncio.gather(rule_task, anomaly_task)   # 2/4 并行

    evidence = extract_evidence(ctx, hits)                      # 3 证据
    verdict_candidate = merge_hits(hits)                        # 5 就高不就低
    llm = await self.agent.adjudicate(LLMInput(ctx, hits, evidence, anomaly))  # 6 裁决
    return self._fuse(verdict_candidate, llm, anomaly, evidence)               # 7 融合
```

`_fuse` 是 V2 方案第五节融合矩阵的代码化，**全部硬约束在此强制执行**：

```python
def _fuse(self, rule_v, llm_v, anomaly, evidence) -> AuditResponse:
    # 硬规则1：涉诈成立 → 风险等级强制高风险（规范）
    if llm_v.is_fraud or rule_v.fraud_candidate:
        risk = "高风险"
    else:
        risk = llm_v.risk_level
    # 硬规则2：规则候选风险 > LLM 风险 → 就高不就低 + 冲突标记
    review, reason = llm_v.needs_review, "none"
    if rule_v.fraud_candidate:
        review, reason = True, "rule_hit_fraud"
    if RISK_ORDER.get(rule_v.candidate_risk, 0) > RISK_ORDER.get(risk, 0):
        risk, review, reason = rule_v.candidate_risk, True, "rule_llm_conflict"
    # 硬规则3：异常分只单向触发复核，不改类型/等级
    confidence = llm_v.confidence
    if anomaly.is_outlier:
        if not llm_v.is_violation and not review:
            review, reason = True, "anomaly_only"
        elif llm_v.is_violation:
            confidence = min(1.0, confidence + 0.05)
    if llm_v.confidence < self.cfg.low_conf_threshold and not review:
        review, reason = True, "low_confidence"
    return AuditResponse(..., risk_level=risk, needs_human_review=review,
                         review_reason=reason, confidence=confidence, anomaly=anomaly)
```

这段代码就是"异常 ≠ 违规"边界的最终防线：`anomaly` 在 `_fuse` 中**没有任何路径**能写 `violation_type` / `risk_level`。

## 十二、实施顺序与每步完成判据

按依赖关系排序（与 V2 方案 Step 编号对应）：

1. **schema + PipelineContext**（Step 1）→ 判据：年报补录案例可手写出一份合法 AuditResponse JSON。
2. **preprocess + normalization.yaml**（Step 3 前半）→ 判据：规范全部典型案例文本可切轮、归一。
3. **chongqing_rules.yaml 首版 + rule_engine + merge**（Step 2/4）→ 判据：13 个规则族中先覆盖引流/贷款/涉诈五类，规范案例单测全绿；这是工作量最大的一步，规则条目预计 40-60 条。
4. **evidence_extractor**（Step 3 后半）→ 判据：每个 RuleHit 能产出原文一致的证据片段。
5. **llm_agent + 提示词 + few_shots**（Step 5）→ 判据：mock 测试全绿；真模型评测脚本对黄金样本可接受。
6. **audit_pipeline + _fuse + FastAPI**（Step 6）→ 判据：curl 一条文本拿到完整响应；此时 `ANOMALY_ENABLED=false`，即 V1 形态 MVP 上线。
7. **feature_extractor + feature_spec.yaml**（Step 8）→ 判据：特征确定性与快照测试通过。
8. **train_anomaly + batch_screen 离线通道**（Step 9）→ 判据：在历史语料上训练出质量达标工件，筛查报告人工抽检合格。
9. **anomaly_service 在线接入 + 融合矩阵生效**（Step 10）→ 判据：黄金样本主判定与第 6 步零差异；杜撰新话术触发 anomaly_only。
10. **复核闭环 SOP**（Step 11）→ 判据：完成一轮"筛查发现→规则回写→回归通过"演练。

第 1-6 步构成可交付的 V1 MVP；7-10 为 PyOD 增量，可在 MVP 上线后并行推进，互不阻塞。

## 十三、实现层面的主要风险点与对策

1. **规则正则的召回/误伤平衡**是首版质量的决定因素：对策是每条规则必须随附正反测试用例（命中样例 + 不应命中样例）才允许合入，`cases.json` 与规则库同 PR 演进。
2. **ASR 文本质量极差**（规范案例可见大量错别字、串音）：正则需容忍变体（如"幺"=1、"企微/企业微信"），归一表会持续膨胀，需把归一表更新纳入与规则相同的回归流程。
3. **LLM JSON 偶发不合规**：三重保障（json mode + Pydantic + sanity check + 重试 + fallback）已覆盖；fallback 结果是"无法判定 + 强制人工复核"，宁可送人工不可错判。
4. **pyod v3 ADEngine API 演进**：用 `ADEngineAdapter` 收敛依赖面；若 ADEngine 不可用，退化方案是直接手工组合 `IForest+ECOD+KNN` 三检测器 + 自实现 rank 共识，功能等价、丢失的只是自动选型与陷阱检查。
5. **超长通话 token 成本**：截断策略（命中轮次±3 + 首尾各5轮）需在评测脚本中验证不丢关键证据；证据提取在截断前完成，保证证据不受截断影响。
