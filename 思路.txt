# 重庆行业卡反诈质检策略 Agent 实施计划

## Summary

- 目标：实现一个**输入为对话文本**的重庆行业卡反诈质检策略 Agent，首期以**离线单条 API** 形式提供能力。
- 方案：采用 **Python + FastAPI + 规则引擎 + LLM 单 Agent** 的组合架构。
- 原则：先用规则做确定性识别、证据抽取和风险初筛，再由 LLM 基于规则与证据做最终归类、风险判定、说明生成和人工复核建议。
- 范围：本期只处理**文本质检**，不接 ASR、音频流、实时流式、前端界面、数据库看板。

## Current State Analysis

### 已探索到的仓库现状

- 当前仓库仅包含需求与思路文档，尚未发现现成代码、服务骨架、测试目录或部署配置。
- 需求主文档：`d:\AI\通话质检agent\重庆行业卡质检复核规范V1.1_20260518.md`
- 可参考思路文档目录：`d:\AI\通话质检agent\docs\ideas`

### 需求文档中可直接落地为规则的关键约束

- 质检结论必须至少包含：**正常/违规**、**风险等级**、**判定说明**。
- 违规场景采用**就高不就低**原则。
- 涉诈识别与普通违规要区分；一旦判定涉诈，风险等级统一按**高风险**处理。
- 输出说明需遵循：**违规/涉诈类型 + 简要分析**。
- 规则并非简单关键词匹配，存在大量**条件组合**与**上下文例外**。

### 文档中最关键的策略难点

#### 1. 同一主题下存在细粒度条件分支

- 例如“引流第三方平台”并不等于所有加微信都违规：
- 外呼人员主动添加用户微信：合规。
- 第三方添加用户微信：贷款场景低风险，其他场景合规。
- 引导用户主动添加微信：高风险。
- 引导下载 APP、点链接、扫二维码、加群、关注服务号：高风险。

#### 2. 风险判定依赖“场景 + 行为 + 对象 + 资金动作”

- 例如贷款相关要区分：
- 合规平台贷款、助贷包装。
- 网贷/小贷平台低风险。
- 不看征信、包下款、保证金、解冻金、提前收费等低风险或高风险场景。
- AB 贷、砍头息、债务优化属于高风险。

#### 3. 需支持“违规”与“涉诈”的二阶段判断

- 先识别违规场景，再识别是否升级为涉诈。
- 典型涉诈类型包括：机票退改签、手机租赁套路贷、证券投资类、网贷平台退息退费、个体工商户年报补录收费等。

#### 4. 证据可解释性是核心

- 规范天然要求人工复核，因此系统输出必须带：
- 命中类型。
- 风险等级。
- 证据片段。
- 判定说明。
- 是否建议人工复核。

### 从 `docs/ideas` 提炼出的可复用方向

- 优先走**单 Agent MVP**，而不是一开始做多 Agent 编排。
- 使用**结构化输出**，禁止自由文本结论。
- 使用**规则优先 + LLM 总结**，而不是把全文直接丢给模型。
- 通过 Few-shot 与知识上下文约束模型，降低“想当然”误判。

## Assumptions & Decisions

### 已锁定的实现决策

- 首期形态：**离线单条 API**
- 架构路线：**规则 + LLM 单 Agent**
- 技术栈：**Python + FastAPI**

### 本计划中的具体假设

- 输入数据已是带说话人标记的对话文本，至少可区分 `left/right` 或 `agent/customer`。
- 首期允许人工提供原始转写文本，不要求接入实时 ASR。
- 首期不依赖数据库，规则、Few-shot、提示词先走本地文件配置。
- 首期以**高可解释、易迭代**优先，不追求最小延迟和最大吞吐。

### 明确不纳入本期

- 音频转写、说话人分离、VAD、流式处理。
- 前端页面、看板、工单系统集成。
- 在线学习、自动回流训练、向量检索平台。
- 多租户、多省份泛化能力。

## Implementation Plan

### Step 1. 先固化统一输入输出协议

- 先定义 API 请求和响应 Schema，避免后续规则与模型结果格式反复变动。
- 验证标准：拿一条“安逸花 + 加微信”的样例，能用固定 JSON 表达完整结论。

### Step 2. 把重庆规范拆成配置化规则

- 将文档中的一级场景、二级子类、风险等级、涉诈标记、例外条件整理进 `chongqing_rules.yaml`。
- 优先做“高价值高确定性”规则，不追求一次覆盖全部长尾。
- 验证标准：规则文件可支持“同主题不同分支”的条件表达。

### Step 3. 实现文本预处理与证据抽取

- 解决原始 ASR 文本格式杂乱问题，统一 speaker、切句、片段引用。
- 验证标准：同一通话可稳定抽出用于判定的关键证据片段。

### Step 4. 实现规则引擎 MVP

- 先实现高精度初筛，不做过深抽象。
- 对明显涉诈、明显违规、明显合规三类先给出稳定候选结论。
- 验证标准：典型规范案例能被规则正确命中大类与基础风险。

### Step 5. 接入 LLM 作为“最终裁决器”

- 模型输入不是原始全文，而是：
  - 规范后的通话文本。
  - 规则命中结果。
  - 证据片段。
  - 少量 Few-shot。
- 模型职责仅限：
  - 细化最终类型
  - 判断是否升级为涉诈
  - 生成“类型 + 分析”说明
  - 给出人工复核建议
- 验证标准：输出稳定为 JSON，且解释不脱离证据。

### Step 6. 实现 FastAPI 单条质检接口

- 提供可直接联调的 HTTP 接口。
- 验证标准：可通过一条文本请求获得完整结构化质检响应。

### Step 7. 用规范文档样例构建回归集

- 从主规范中提取至少 10-20 条代表性案例作为 fixtures。
- 验证标准：每次修改规则后，可快速回归重点场景。

## Verification Steps

### 功能验证

- 用规范中的典型案例逐条验证大类识别是否正确。
- 验证“违规但不涉诈”与“直接涉诈”的边界是否正确。
- 验证“引流第三方平台”中不同微信场景是否能区分。

### 稳定性验证

- 验证同一输入多次调用时输出结构是否稳定。
- 验证未命中明显规则时是否回退为“需人工复核”，而不是强行下结论。

### 可解释性验证

- 每条结论都必须能回溯到原文证据。
- 每条说明都必须可对应规范中的规则描述。

### 回归验证

- 建立最少 10 条黄金样本。
- 后续每次规则改动必须跑回归，重点覆盖：
  - 加微信分支判断
  - 贷款相关细分类
  - 年报补录收费
  - 证券投资
  - 手机租赁套路贷

## Risks

- 最大风险不是模型能力，而是**业务规则细粒度例外很多**，如果直接全文交给模型，误判率会很高。
- 仅靠关键词会把“加微信”类场景大量误伤，必须引入角色、动作方向、业务场景联合判断。
- 如果不做标准样本回归，后续改规则很容易修一处坏一片。

## Recommended Execution Order

1. 建 Schema 与输出协议。
2. 整理 `chongqing_rules.yaml` 第一版。
3. 实现 `preprocess.py`、`rule_engine.py`、`evidence_extractor.py`。
4. 实现 `llm_agent.py` 与 `audit_pipeline.py`。
5. 暴露 FastAPI 接口。
6. 补测试样本与回归测试。

## Success Criteria

- 输入一段重庆行业卡对话文本，可返回稳定 JSON 结果。
- 能区分正常、违规、涉诈三类结果。
- 能输出风险等级、类型、说明、证据、人工复核建议。
- 对规范中的典型高风险与涉诈案例具备可接受的首版识别能力。
